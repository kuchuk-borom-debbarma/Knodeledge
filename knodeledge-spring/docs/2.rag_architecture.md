# RAG Architecture

## Boundaries

Knodeledge v1 is intentionally a personal note RAG system. It does not build a graph, extract
entities, infer domains, or require users to organize information. The only durable user-created
knowledge object is a raw note.

## Storage

Postgres is the authoritative store. Flyway creates:

- `app_users` for dev authentication.
- `notes` for raw source text and indexing state.
- `note_chunks` for chunk text, embeddings, full-text vectors, and metadata.

Notes are soft deleted through `deleted_at`. Chunks are derived data and may be hard deleted when a
note is edited, deleted, or reindexed.

## Retrieval

Chat retrieval runs both search modes for the current user:

- Dense search embeds the question and returns the top 30 chunks by pgvector cosine similarity.
- Sparse search uses `websearch_to_tsquery` and returns the top 30 chunks by full-text rank.
- Candidates are merged by chunk id and scored with a weighted hybrid score.
- Up to 50 candidates are sent to the reranker.
- The top 10 reranked chunks are sent to the answer model.

If the reranker fails, the weighted hybrid score is used as the fallback ordering.

## Answering

The answer model receives only the question and selected chunks. It must return structured output
with answer text, citation chunk ids, and an inference flag. The service validates that all citation
ids appear in the supplied context. Missing or invalid citations become a not-enough-info response.

## Security

The v1 bearer token is the user id returned by login. Controllers resolve it through `CurrentUser`
and pass the resolved UUID to services. SQL reads and writes use explicit `user_id` filters, and
retrieval joins chunks to active notes owned by the same user.
