# Chunking And Indexing

## Note Save

Creating or editing a note writes raw text first and returns without waiting for AI calls. The note
is marked `pending`, old chunks are removed on edit, and a background job receives the note id plus
the expected `index_version`.

## Chunking

The chunker targets 400-800 estimated tokens. It prefers stable note boundaries in this order:

1. Headings.
2. Bullets and numbered list items.
3. Paragraphs.
4. Token fallback for large blocks.

Chunks use a 100-token overlap and include note title plus created date in the stored chunk text.

## Async Indexing

The indexer transitions `pending` to `indexing` only when the stored `index_version` still matches
the queued version. This prevents stale jobs from overwriting newer edits.

For each chunk, the indexer generates an embedding, validates the configured embedding dimension,
stores the vector, and lets Postgres generate `search_text`. A successful run marks the note
`ready`; failures mark the note `failed` with a visible error message.

## Retry

`POST /api/v1/notes/{id}/reindex` increments `index_version`, clears the error, deletes derived
chunks, and queues a fresh indexing job.
