# Hierarchy Architecture

**Status:** Implemented
**Updated:** 2026-06-15
**Area:** Knowledge Model, Retrieval, Mutation, and Exploration

## Design Goals

Knodeledge stores knowledge in the same recursive shape users browse and the model retrieves.

The design must:

1. Keep one authoritative representation.
2. Support arbitrary useful topic depth.
3. Preserve precise facts and conditional meaning.
4. Route without loading all knowledge into model context.
5. Apply local changes without damaging unrelated branches.
6. Keep repeated semantic facts synchronized.
7. expose one stable level at a time to the UI.

## Tree Semantics

Every boundary is one rooted tree. A node has exactly one structural parent. Repeated semantic
knowledge is represented by multiple placements sharing one semantic key.

```text
ROOT
└─ TOPIC
   ├─ ENTITY
   ├─ FACT
   │  └─ CONDITION
   └─ TOPIC
      └─ FACT
```

Structural parenthood answers "where should this knowledge be found?" The typed relation answers
"how does this child relate to this parent?"

Summaries belong to nodes. Relations belong to parent-child links through
`relationToParent`.

## Node Contract

```json
{
  "id": "node_4f8...",
  "semanticKey": "fact:kuku:plays_rainbow_six_siege",
  "parentId": "node_gaming",
  "kind": "FACT",
  "relationToParent": "PLAYS",
  "name": "Rainbow Six Siege",
  "statement": "Kuku plays Rainbow Six Siege.",
  "summary": "A tactical shooter Kuku actively plays."
}
```

### Identity

`id` identifies one placement. The server creates IDs for new nodes and does not trust model-created
persistent IDs.

`semanticKey` identifies meaning across placements. It is stable, lowercase, and descriptive.

Examples:

```text
root:boundary-123
topic:kuku:interests:gaming
fact:kuku:plays_rainbow_six_siege
condition:kuku:drinking_alcohol
```

### Kinds

`ROOT`:

- exactly one per boundary
- parent and relation are null
- name is the boundary subject
- cannot be moved, deleted, or semantically replaced

`TOPIC`:

- groups coherent descendants
- may contain topics, entities, facts, or conditions when semantically valid
- statement may be null

`ENTITY`:

- represents a named concept or object
- statement may be null when summary is sufficient

`FACT`:

- stores one precise durable statement
- statement is mandatory
- may contain conditions

`CONDITION`:

- stores one precise qualification
- statement is mandatory
- may contain nested conditions

### Relations

Relations are validated `UPPER_SNAKE_CASE` values. The model may choose domain-specific values, but
these common relations should be preferred:

| Relation | Use |
|---|---|
| `CONTAINS` | Topic grouping |
| `IS_A` | Classification |
| `HAS_ROLE` | Occupation or role |
| `LIKES` | General preference |
| `PLAYS` | Instrument or game activity |
| `WATCHES` | Viewed media |
| `CREATES` | Creative activity |
| `HAS_TRAIT` | Behavioural trait |
| `HAS_STATE` | Physical or emotional state |
| `HAS_PREFERENCE` | Preference fact |
| `WHEN` | Trigger condition |
| `ALL_OF` | Required conjunction |
| `ANY_OF` | Alternative condition |
| `NOT` | Negated condition |

## Hierarchy Contract

```json
{
  "boundaryId": "boundary-123",
  "rootNodeId": "root_boundary123",
  "nodes": [],
  "version": 7
}
```

The repository stores the complete validated value. API level responses are derived views and are
not separately persisted.

Version starts at one and increments once for every applied patch or rebalance patch.

## Level Projection

The normal read API returns one local projection:

```text
current       selected node
children      immediate children only
breadcrumbs   root-to-current path
leaf          true when current has no children
version       current hierarchy version
```

The projection intentionally excludes:

- grandchildren
- sibling branches outside the breadcrumb path
- full semantic-copy locations
- unrelated facts

This gives predictable UI payload size regardless of total hierarchy size.

## Routing

Routing begins with the root as the current selection.

At each step:

1. Read immediate children of active selections.
2. Send current-node and candidate-child metadata to the model.
3. Let the model stop at the current level for broad questions.
4. Otherwise keep at most two highest-scored valid children.
5. Preserve selected leaves as terminal results.
6. Continue through non-leaf selections.

Only IDs returned from the supplied candidate set are accepted.

Candidate metadata contains name, kind, relation, semantic key, and summary. Exact statements are
withheld until the selected answer or patch context requires them.

## Local Slice

Mutation uses a local slice containing:

- every selected node
- every root-to-selected ancestor
- immediate children of selected nodes
- every placement matching extracted semantic keys
- paths needed to update summaries for those matching placements

The model sees only this slice. The server still applies the validated patch against the complete
in-memory hierarchy.

## Patch Contract

```text
HierarchyPatch {
  addNodes
  updateSemanticNodes
  moveNodes
  deleteSemanticKeys
  ancestorSummaries
}
```

### Additions

Model additions use temporary IDs and parent references. The server maps temporary IDs to generated
persistent IDs.

An addition parent must be:

- a routed existing node, or
- another addition in the same patch

This supports adding complete nested branches in one atomic operation.

### Semantic Updates

Updates target `semanticKey`, not placement ID. The processor updates kind, name, statement, and
summary on every matching placement.

Parent and relation remain placement-specific.

### Moves

Moves target one placement ID. The node must belong to the routed slice. The destination must be a
routed existing parent or a temporary topic added by the same patch. The root cannot move.

### Deletions

Deletes target semantic keys. At least one matching placement must be present in the routed slice.
Every matching placement and its condition descendants are removed.

### Ancestor Summaries

Every changed parent and all of its ancestors through the root require a supplied fresh summary.
The patch is rejected when any required summary is missing.

New nodes already carry their own summaries and therefore do not need separate summary updates.

## Semantic Copies

Multiple placements are allowed because browsing is tree-shaped while knowledge may be relevant in
multiple topics.

Example:

```text
Profile
└─ Plays guitar [semanticKey=fact:kuku:plays_guitar]

Interests
└─ Music
   └─ Plays guitar [semanticKey=fact:kuku:plays_guitar]
```

Both placements must have identical:

- kind
- name
- statement
- summary

They may differ in:

- ID
- parent
- relation to parent

Patch application synchronizes content before final validation.

## Conditional Knowledge

Conditional meaning remains traversable and visible:

```text
Gets tipsy [FACT, HAS_STATE]
└─ Drinking alcohol [CONDITION, WHEN]
```

Conjunction:

```text
Likes this game type [FACT, HAS_PREFERENCE]
├─ Manageable by Kuku [CONDITION, ALL_OF]
└─ Has NPC village [CONDITION, ALL_OF]
```

Alternative:

```text
Enjoys strategy games [FACT]
├─ Village management [CONDITION, ANY_OF]
└─ Colony management [CONDITION, ANY_OF]
```

Negation:

```text
Enjoys a game [FACT]
└─ Requires aggressive monetization [CONDITION, NOT]
```

Conditions use precise statements so relation labels never carry the entire meaning.

## Rebalancing

The system permits deep detail and does not impose a small product depth limit. Depth 32 is a
technical guard against runaway output and cycles.

A parent with more than twelve children is eligible for local rebalancing. The model receives only:

- overloaded parent
- immediate children
- breadcrumb path

It can create topic groups and move children beneath them. It cannot delete facts or change their
semantic content.

Useful grouping is preferred over equal group size. A coherent level may remain above the threshold
when extra grouping would obscure meaning.

## Atomicity

An ingestion request runs under one boundary lock:

1. Read current hierarchy.
2. Extract and route.
3. Build and validate patch.
4. Apply semantic synchronization.
5. Rebalance affected levels.
6. Validate final hierarchy.
7. Save final hierarchy.

No intermediate hierarchy version is exposed.

## Validation Algorithm

Validation builds an ID map and verifies:

1. Boundary ID, root ID, nodes, and version exist.
2. IDs are unique.
3. Exactly one parentless node exists.
4. Root ID points to that `ROOT` node.
5. Every non-root node has an existing parent.
6. Every parent chain reaches root without a cycle.
7. No chain exceeds depth 32.
8. Required fields are nonblank.
9. Fact and condition statements are nonblank.
10. Relations match `[A-Z][A-Z0-9_]*`.
11. Semantic copies agree.

Patch guards additionally verify local authority and root immutability.

## Failure Behaviour

Model failures, invalid routes, invalid patches, missing summaries, or final validation errors abort
the operation. No hierarchy save occurs.

Prompt-cache failures are removed from cache so future requests can retry.

API errors use normal Spring error handling. The frontend retains its last valid level and displays
an error toast.

## Storage and Migration

Current persistence is in memory. `HierarchyRepository` is the only active write store.

`LegacyGraphRepository` is read only from the hierarchy workflow's perspective and exists solely
for a one-time conversion when hierarchy data is absent.

Once converted, all reads and writes use `BoundaryHierarchy`.

## API Examples

Root:

```http
GET /api/v1/hierarchy/boundary-123/level?userId=user-1
```

Nested level:

```http
GET /api/v1/hierarchy/boundary-123/level?userId=user-1&nodeId=node_gaming
```

Debug export:

```http
GET /api/v1/hierarchy/debug/boundary-123?userId=user-1
```

The debug route is intentionally the only normal HTTP operation returning the full hierarchy.

## Frontend Navigation

Each level is drawn as a centered radial projection:

- current node at `(0, 0)`
- children positioned on a deterministic ring
- relation labels on directed edges
- topic nodes visually distinct from entities, facts, and conditions
- clicked child becomes next current node
- breadcrumb click loads any ancestor
- back loads the immediate parent

The UI does not infer hierarchy structure. It renders the server's level contract.

## Test Matrix

Deterministic tests cover:

- detailed Kuku hierarchy
- immediate-child level isolation
- breadcrumb generation
- leaf statements
- semantic-copy synchronization
- unrelated-branch preservation
- nested conditional additions
- root mutation rejection
- unrelated delete rejection
- cycle and depth rejection
- overloaded-level detection

Provider-backed evaluation should additionally cover routing quality, semantic grouping, broad
question stopping, precise leaf selection, and migration fidelity.
