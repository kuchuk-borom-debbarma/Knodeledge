# Prompt Engineering Rules

**Status:** Active
**Updated:** 2026-06-15
**Area:** Hierarchy Ingestion, Routing, Rebalancing, Migration, and Answering

## Principles

Prompts propose structured intent. Java code owns authority.

Every hierarchy prompt must:

1. Request a typed response DTO.
2. Define allowed IDs and local context.
3. Preserve precise factual meaning.
4. Avoid inventing knowledge.
5. Keep summaries useful for future routing.
6. Leave validation and persistence to deterministic code.

## Prompt Stages

| Resource | Purpose |
|---|---|
| `01_extract_knowledge.st` | Split notes into precise hierarchy items |
| `02_route.st` | Select up to two children at one hierarchy level |
| `03_patch.st` | Produce local additions, updates, moves, deletes, and summaries |
| `04_rebalance.st` | Group one overloaded level |
| `05_answer.st` | Answer from selected paths and statements |
| `06_migrate_legacy_graph.st` | Perform one-time conversion from retired data |

## Knowledge Extraction

Extraction returns `KnowledgeExtraction`.

Each item needs:

- stable semantic key
- concise name
- exact statement for facts and conditions
- routing summary
- node kind
- typed relation hint
- nested child conditions when present

### Atomicity

One item should express one durable claim.

Good:

```text
Kuku plays guitar.
Kuku sometimes makes music.
Kuku watches Attack on Titan.
```

Bad:

```text
Kuku likes music, anime, games, programming, and many other things.
```

### Precision

Preserve:

- subject
- predicate
- object
- negation
- degree
- frequency
- temporal or situational condition

For example, "does not get angry easily" must not become "never gets angry."

### Conditions

Conditions are nested items. Use relation hints:

- `WHEN`
- `ALL_OF`
- `ANY_OF`
- `NOT`

Do not flatten conditions into summary text.

## Routing

The router sees current nodes and immediate candidate children only.

It returns:

```text
RouteResponse {
  stopAtCurrent
  choices
}
```

Rules:

- choose only supplied child IDs
- return at most two choices
- use `stopAtCurrent` for broad questions answerable from current summaries
- prefer specific children when precise knowledge is requested
- permit two branches for cross-topic notes or questions
- score choices consistently from zero to one

Java discards unknown IDs, sorts by score, and enforces beam width two.

## Patch Generation

Patch generation receives extracted knowledge and routed local slices.

It returns:

```text
HierarchyPatch {
  addNodes
  updateSemanticNodes
  moveNodes
  deleteSemanticKeys
  ancestorSummaries
}
```

### Add Nodes

Use temporary IDs. A parent reference must identify:

- an existing node in the supplied slice, or
- another temporary node in the response

Return parents before children for readability.

Create topic layers when they improve navigation. Do not create empty decorative layers.

### Update Semantics

Use semantic updates when existing meaning changes or gains precision. Do not add a second semantic
key for paraphrased existing knowledge.

### Duplicate Placement

Duplicate only when one fact is genuinely useful in multiple branches. Use:

- distinct temporary placement IDs
- the same semantic key
- identical kind, name, statement, and summary

### Move Nodes

Move only clearly misplaced nodes supplied in local context. `newParentRef` may reference a routed
existing node or an addition's temporary ID. Never restructure unrelated branches.

### Delete Knowledge

Delete only when the note explicitly retracts, corrects, or invalidates stored knowledge. Absence
from a new note is never deletion evidence.

### Summaries

Refresh every affected existing parent and all ancestors through root.

Summary quality:

- concise enough for routing
- specific enough to distinguish siblings
- truthful after the patch
- free of unsupported details
- useful without reading descendants

Missing required ancestor summaries cause deterministic rejection.

## Rebalancing

Rebalancing receives one parent, immediate children, and breadcrumbs.

It may:

- create coherent topic nodes
- move existing children below new topics
- refresh ancestor summaries

It may not:

- delete knowledge
- alter factual meaning
- move the level root
- reference nodes outside supplied context

Set `changeNeeded` false when the current level remains clearer than additional grouping.

## Answering

Answer context contains selected nodes and root-to-node paths.

Internal summaries support broad questions. Exact fact and condition statements support precise
questions.

Answers must:

- use only supplied context
- preserve conditions and negation
- distinguish known from unknown
- state insufficiency directly
- remain concise

## Migration

Migration is the only prompt allowed to inspect the complete retired data snapshot.

It must:

- use the supplied boundary ID
- create exactly one root
- preserve every durable fact
- convert qualified meaning into condition descendants
- create coherent progressive topics
- keep semantic duplicates consistent
- return hierarchy version one

The resulting hierarchy is fully validated before save.

## Context Size Rules

Steady-state prompts must not serialize `BoundaryHierarchy`.

Allowed context:

- one candidate level for routing
- routed paths and immediate children for patching
- one overloaded level for rebalancing
- selected paths for answering

Debug APIs do not feed full export into normal prompts.

## Prompt Cache

Prompts are cacheable only by exact identity. Changes to any system prompt, local slice, query,
model, stage, or response DTO produce a different key.

Do not include timestamps, random IDs, or irrelevant volatile fields in prompts. They reduce safe
cache reuse.

## Structured Output Safety

DTO fields should remain explicit and shallow enough for model reliability.

Java must handle:

- null lists as empty
- null response as failure
- blank text response as failure
- invalid IDs as rejection
- missing summary refreshes as rejection
- invalid relations as rejection
- root mutation as rejection

Never repair an unsafe structural response by silently broadening model authority.

## Evaluation Scenarios

Provider-backed prompt evaluation should include:

1. Kuku note containing guitar, music creation, anime, gaming, programming, temperament, and alcohol.
2. Cross-topic note selecting at most two branches per routing level.
3. Conditional preference with `ALL_OF` children.
4. Negated behavioural fact.
5. Broad question stopping at `Interests`.
6. Precise question reaching `Rainbow Six Siege`.
7. Duplicate guitar placement under Profile and Music.
8. Explicit correction updating all duplicate placements.
9. Overloaded gaming level grouped without lost facts.
10. Hallucinated route IDs and unrelated patch references rejected.
